### **Название задачи:** Реализация просмотра истории заказов покупателя

### **Автор:** команда NovaMarket

### **Дата:** 2026-06-30

### **Функциональные требования**

Покупатель после завершения доставки заходит в личный кабинет и работает с историей
своих заказов.

| № | Действующие лица или системы | Use Case | Описание |
| :-: | :- | :- | :- |
| 1 | Покупатель, Order History Service | Просмотр списка заказов | Покупатель открывает «Мои заказы» и видит список всех заказов: стоимость, дату оформления, способ доставки, финальный статус |
| 2 | Покупатель, Order History Service | Просмотр деталей заказа | Покупатель открывает заказ и видит состав (товары, количество, цена), итоговую сумму, способ доставки и оплаты, финальный статус (доставлен/отменён) |
| 3 | Покупатель, Order History Service, Payment Service | Скачивание чека | Покупатель скачивает чек по оплаченному заказу |
| 4 | Покупатель, Catalog/Review Service | Оставить отзыв | Покупатель оставляет отзыв о товаре из заказа |
| 5 | Покупатель, Cart/Order Service | Заказать снова | Покупатель повторяет заказ — товары из истории добавляются в новую корзину и проходят обычный сценарий оформления |

### **Нефункциональные требования**

| № | Требование |
| :-: | :- |
| 1 | Время отклика всех пользовательских интерфейсов ≤ 1 c в 99,99% запросов; 0,3 c в 98% запросов |
| 2 | Чтение истории не должно деградировать под нагрузкой до 40 000+ заказов в день и не нагружать write-модель оформления заказа |
| 3 | Данные истории по завершённому заказу неизменяемы (immutable) — заказ доставлен/отменён |
| 4 | Масштабируемость read-нагрузки независимо от write-нагрузки |
| 5 | Лёгкость добавления новых полей в карточку истории без изменения сервисов-источников |
| 6 | Согласованность данных истории — eventual consistency допустима (история показывается уже после завершения заказа) |

### **Решение**

Выбран паттерн **CQRS** (Command Query Responsibility Segregation): разделяем модель
записи (оформление заказа) и модель чтения (история заказов).

Вводится новый сервис **Order History Service** (query-сторона) со своей
**read-моделью** — денормализованным хранилищем `Order History Read DB`
(PostgreSQL с денормализованной таблицей или Elasticsearch). Сервис подписан на
доменные события из Kafka (`OrderConfirmed`, `OrderCancelled`, `PaymentSucceeded`,
`PaymentRefunded`, `DeliveryRequestCreated` и изменения статуса) и на их основе строит
готовые проекции карточек заказов. Запрос истории обслуживается **одним чтением** из
read-модели, без обращения к другим сервисам в момент запроса.

Диаграммы C4:
- Контекст (C1): [`c4-context.puml`](./c4-context.puml) → `images/NovaMarket-C4-Context.png`
- Контейнеры (C2): [`c4-container.puml`](./c4-container.puml) → `images/NovaMarket-C4-Container-OrderHistory.png`

**Логика выбора:**
- Ключевое НФТ — строгий отклик (≤ 1 c / 0,3 c в 98%) при росте нагрузки. Денормали­
  зованная read-модель отдаёт список и детали заказа за один индексированный запрос,
  что надёжно укладывается в SLA даже на пике.
- Данные завершённого заказа неизменяемы, поэтому проекции не требуют сложной
  инвалидации — их можно безопасно кэшировать и реплицировать.
- Read- и write-нагрузка масштабируются независимо: тяжёлые чтения истории не влияют
  на оформление заказов (Order DB).
- Read-модель уже подписана на события Saga из Task1 — переиспользуем существующий
  поток событий, не добавляя синхронных вызовов между сервисами.
- Новые поля в карточке истории добавляются изменением только проекции Order History
  Service, без правок сервисов-источников.

### **Альтернативы**

**1. API Composition.** Сервис-агрегатор в момент запроса синхронно опрашивает Order,
Payment, Delivery, Catalog и склеивает ответ.
- Плюсы: нет дублирования данных, всегда «свежие» данные, минимум новой инфраструктуры.
- Минусы: задержка = сумма/максимум вызовов нескольких сервисов + сетевые издержки —
  трудно гарантировать ≤ 1 c под нагрузкой; fan-out усиливает связанность и риск
  каскадных отказов; чтения истории нагружают операционные БД сервисов. Не проходит по
  НФТ-1 и НФТ-2.

**2. Event Sourcing.** Хранить заказ как поток событий и восстанавливать состояние
истории из лога.
- Плюсы: полный аудит, естественная история изменений, возможность пересборки проекций.
- Минусы: чтение требует свёртки потока (медленно без отдельных проекций) — то есть всё
  равно нужен CQRS поверх; высокая сложность внедрения и эксплуатации, перепроектиро­
  вание всех сервисов записи. Избыточно для задачи показа истории.

**3. Комбинация CQRS + Event Sourcing.** Рассматривалась как развитие: ES для Order
Service + CQRS-проекции для истории. Отложено — даёт аудит, но не оправдывает рост
сложности на текущем этапе. CQRS закрывает требование самостоятельно.

### **Недостатки, ограничения, риски**

- **Eventual consistency.** Между завершением заказа и появлением в истории есть лаг
  (обработка события). Допустимо: история нужна уже после завершения заказа.
- **Дублирование данных.** Часть данных заказа хранится и в источниках, и в read-модели
  — рост объёма хранилища и необходимость следить за схемой проекций.
- **Зависимость от шины событий.** Потеря/задержка событий ⇒ пропуски в истории.
  Требуются гарантии доставки (at-least-once), идемпотентные апдейты проекций по ключу
  заказа, ретраи и контроль consumer lag.
- **Пересборка проекций.** При изменении формата карточки нужен механизм rebuild
  (переигрывание событий или backfill из источников) — стоит заложить заранее.
- **Эксплуатация.** Дополнительный сервис и хранилище (мониторинг lag, алерты на отста­
  вание read-модели, согласование версий событий).

## Часть 1. Событийная архитектура

### Выбранные микросервисы

| Сервис | Назначение |
| --- | --- |
| **API Gateway (NGINX)** | Единая точка входа: маршрутизация, rate limiting, прокси к сервисам |
| **Catalog Service** | Каталог товаров, фильтрация, карточки, отзывы; источник актуальных цен и наличия |
| **Cart Service** | Корзина покупателя (без оформления заказа) |
| **Order Service** | Оформление заказа, хранение и обновление статусов, участник Saga |
| **Inventory Service** | Проверка наличия и резервирование товара на складе |
| **Payment Service** | Оплата, привязка карты, автосписания, возвраты |
| **Delivery Service** | Формирование заявки на доставку, трек-номер, интеграция с логистом |
| **Notification Service** | Push-уведомления покупателю (статусы) и продавцу (новые заказы) |
| **Message Broker (Kafka)** | Шина событий между сервисами |

Каждый сервис владеет своей БД (database per service). Каталог и статус заказа
читаются синхронно (REST), что обеспечивает требование к быстрому отклику; тяжёлые
шаги оформления (резерв, оплата, доставка) выполняются асинхронно через события.

### Способ отрисовки потоков событий

Выбран **вариант 1 — через брокер (топики)**, потому что в решении есть события,
которые потребляют сразу несколько сервисов:

- `PaymentSucceeded` слушают **Delivery**, **Notification** и **Order**;
- `DeliveryRequestFailed` слушают **Payment**, **Inventory** и **Order**.

При таком fan-out схема «через брокер» нагляднее, чем прямые стрелки между сервисами.

## Часть 2. Saga оформления заказа (хореография)

Оформление заказа — распределённая транзакция из нескольких шагов (резерв → оплата →
доставка). Выбрана **хореография**: нет центрального оркестратора, каждый сервис
реагирует на событие предыдущего шага и публикует свой результат. При ошибке
запускаются компенсирующие события, откатывающие уже выполненные шаги.

Полный список событий с типами (`domain` / `failure` / `compensation`), издателями
и подписчиками — в [`events-registry.md`](./events-registry.md).

### Сценарии

- **Успешный** ([`saga-order-success.puml`](./saga-order-success.puml)):
  `OrderCreated` → `InventoryReserved` → `PaymentSucceeded` →
  `DeliveryRequestCreated` → `OrderConfirmed`.
- **Ошибочные** ([`saga-order-failures.puml`](./saga-order-failures.puml)) — три
  ветки:
  1. нет товара → `InventoryReservationFailed` → `OrderCancelled`;
  2. ошибка оплаты → `PaymentFailed` → `InventoryReleased` → `OrderCancelled`;
  3. ошибка доставки → `DeliveryRequestFailed` → `PaymentRefunded` +
     `InventoryReleased` → `OrderCancelled`.
